## 2. ЦЕЛИ И НАЗНАЧЕНИЕ СОЗДАНИЯ СИСТЕМЫ

### 2.1 Цели создания АИС

| # | Цель | Критерий достижения |
|---|------|-------------------|
| 1 | Централизовать учёт оценок | 100% оценок фиксируются в единой БД, бумажные журналы не требуются |
| 2 | Обеспечить студентам прозрачность успеваемости | Студент видит все свои оценки и историю изменений в режиме реального времени |

### 2.2 Назначение АИС

АИС «Успеваемость» предназначена для автоматизации процесса регистрации, хранения, контроля и анализа текущей академической успеваемости студентов: ввода оценок преподавателями, просмотра успеваемости студентами и формирования сводной статистики для деканата.

## 3. ХАРАКТЕРИСТИКА ОБЪЕКТА АВТОМАТИЗАЦИИ

### 3.1 Описание объекта

Объект автоматизации — процессы учёта и контроля текущей успеваемости студентов учебного заведения, включающие:
- Регистрацию результатов текущей, рубежной и итоговой аттестации
- Ведение учёта посещаемости занятий
- Формирование рейтинга студентов по дисциплинам
- Контроль успеваемости со стороны деканата
- Документирование замечаний и комментариев преподавателей

### 3.2 Существующее состояние автоматизации (As-Is)

Учёт успеваемости ведётся преимущественно в бумажных журналах или разрозненных таблицах Excel. Типичные проблемы:
- Студент не имеет оперативного доступа к своим оценкам
- Нет единого хранилища: у каждого преподавателя свой файл
- Отсутствует история изменений оценок
- Деканат получает сведения об успеваемости с задержкой
- Высока вероятность потери данных при утере файлов/тетрадей

### 3.3 Условия эксплуатации

- Система работает в режиме 24/7 в период учебного семестра
- Доступ через веб-браузер по локальной сети вуза или сети Интернет
- Количество одновременных пользователей: до 300 человек
- Хранение данных: сервер учебного заведения или облачная инфраструктура

### 3.4 Пользователи системы

| Категория | Кол-во | Роль в системе |
|-----------|--------|---------------|
| Студенты | до 250 | Просмотр своих оценок, посещаемости и рейтинга |
| Преподаватели | до 30 | Ввод и редактирование оценок, добавление комментариев |
| Администраторы (деканат) | до 5 | Управление системой, пользователями, просмотр аналитики |

## 4. ТРЕБОВАНИЯ К АВТОМАТИЗИРОВАННОЙ СИСТЕМЕ

### 4.1 Требования к структуре АС

#### 4.1.1 Состав подсистем

| Подсистема | Назначение |
|-----------|-----------|
| Подсистема аутентификации | Регистрация, вход, управление сессиями (JWT) |
| Подсистема управления оценками | Ввод, редактирование, просмотр оценок студентов |
| Подсистема комментирования | Обратная связь преподавателя к студенту по оценке |
| Подсистема справочников | Управление дисциплинами и типами оценок |
| Подсистема администрирования | Управление пользователями и ролями |
| Подсистема аналитики и отчётности | Дашборд и статистика успеваемости |

#### 4.1.2 Архитектура системы

Система реализуется по клиент-серверной архитектуре:

```
┌─────────────────┐     HTTP/REST     ┌──────────────────┐
│  Nuxt 3 SPA     │ ◄────────────────►│  Go REST API     │
│  (браузер)      │   JSON / JWT      │  (Gin framework) │
└─────────────────┘                   └────────┬─────────┘
                                               │ SQL
                                      ┌────────▼─────────┐
                                      │   PostgreSQL      │
                                      └──────────────────┘
```

#### 4.1.3 Информационное взаимодействие

- Frontend ↔ Backend: REST API, формат JSON, аутентификация Bearer JWT
- Backend ↔ БД: raw SQL через `database/sql` + драйвер `lib/pq`, соединение по TCP

### 4.2 Требования к функциям (задачам), выполняемым АС

#### 4.2.1 Подсистема аутентификации

| Функция | Результат | Требование |
|---------|-----------|-----------|
| Регистрация пользователя | Учётная запись создана | Email уникален, пароль ≥ 6 символов, хэширование bcrypt |
| Вход в систему | Получен JWT Access + Refresh токен | Access-токен живёт 15 минут, Refresh — 30 дней |
| Обновление токена | Новый Access-токен | Проверка подписи и срока действия Refresh-токена |
| Выход из системы | Refresh-токен инвалидирован | Токен удаляется из БД |

#### 4.2.2 Подсистема управления оценками

| Функция | Результат | Требование |
|---------|-----------|-----------|
| Создание записи об оценке | Оценка зарегистрирована с уникальным номером | Обязательные поля: название работы, дисциплина, тип оценки, студент |
| Просмотр списка оценок | Пагинированный список с фильтрами | Фильтры: дисциплина, тип, статус аттестации; страница по 20 записей |
| Просмотр детальной записи | Все данные оценки + история изменений | Студент видит только свои оценки, преподаватель — по своим дисциплинам |
| Выставление оценки | Балл сохранён, запись переходит в статус «оценено» | Только преподаватель; балл в диапазоне 0..max_score дисциплины |
| Редактирование оценки | Балл обновлён, изменение занесено в историю | Только преподаватель-автор с обязательным комментарием-обоснованием |
| Финальная аттестация | Статус «аттестован» / «не аттестован» | По итогам рубежного/итогового типа оценки |

#### 4.2.3 Подсистема комментирования

| Функция | Результат | Требование |
|---------|-----------|-----------|
| Добавление комментария к оценке | Комментарий сохранён с автором и временем | Доступно и студенту и преподавателю |
| Внутренняя заметка преподавателя | Заметка сохранена и скрыта от студента | Только преподаватель; флаг `is_internal = true` |

#### 4.2.4 Подсистема администрирования

| Функция | Результат | Требование |
|---------|-----------|-----------|
| Управление пользователями | Изменение роли / статуса | Только admin; нельзя заблокировать себя |
| Управление дисциплинами | CRUD операции | Нельзя удалить дисциплину с существующими оценками |
| Управление типами оценок | CRUD операций | Типы: Текущая, Рубежная, Итоговая; задаётся max_score |
| Просмотр дашборда | Статистика системы | Данные: средний балл по дисциплинам, % неаттестованных, топ-5 преподавателей по активности |

### 4.3 Требования к видам обеспечения АС

#### 4.3.1 Информационное обеспечение

- СУБД: **PostgreSQL 15+**
- Драйвер: **lib/pq** (Go), raw SQL запросы с параметризацией `$1, $2, ...`
- Формат обмена данными: **JSON (UTF-8)**
- Структура БД описана в разделе аналитических материалов (файл `03_analytics.md`)
- Резервное копирование БД: ежедневно, хранение 7 резервных копий

#### 4.3.2 Программное обеспечение

**Backend:**
- Язык: **Go 1.21+**
- HTTP framework: **Gin v1.9+**
- БД-драйвер: **lib/pq** (чистый SQL, без ORM)
- JWT: **golang-jwt/jwt v5**
- Хэширование паролей: **bcrypt** (cost factor ≥ 12)
- Переменные окружения: **godotenv**

**Frontend:**
- Язык: **TypeScript 5+**
- Framework: **Nuxt 3** (SSR отключён, режим SPA)
- Стейт-менеджмент: **Pinia 2**
- HTTP-клиент: **Axios**
- UI-библиотека: **Naive UI**
- Стили: **SCSS** (с адаптивными миксинами)

**Инфраструктура:**
- Контейнеризация: **Docker + Docker Compose** (опционально)
- ОС сервера: Linux (Ubuntu 22.04 LTS)

#### 4.3.3 Техническое обеспечение

**Минимальные требования к серверу:**

| Параметр | Значение |
|---------|---------|
| CPU | 2 ядра, 2 GHz |

| Браузер | Chrome 90+, Firefox 88+, Safari 14+, Edge 90+ |
| Разрешение | от 390px (мобильные) до 1920px (desktop) |
| JavaScript | Включён |

#### 4.3.4 Лингвистическое обеспечение

- Интерфейс системы: **русский язык**
- API-идентификаторы (поля, эндпоинты): **английский язык**
- Кодировка: **UTF-8**

### 4.4 Общие технические требования

#### 4.4.1 Требования к надёжности

| Показатель | Значение |
|-----------|---------|
| Доступность (uptime) | ≥ 99% (не более 7 часов простоя в месяц) |
| Время восстановления после сбоя | ≤ 1 час |
| Сохранность данных при сбое | Транзакционная целостность PostgreSQL |

#### 4.4.2 Требования к безопасности и защите информации

- Пароли хранятся только в виде хэша bcrypt, не передаются в открытом виде
- HTTPS обязателен в продуктивной среде (TLS 1.2+)
- JWT Access-токен содержит только ID и роль пользователя, персональные данные не хранятся в токене
- Все API-запросы (кроме auth) требуют валидного Access-токена
- RBAC (Role-Based Access Control): каждый эндпоинт проверяет роль пользователя
- SQL-инъекции предотвращаются параметризованными запросами (`$1, $2, ...`)
- XSS предотвращается экранированием выводимых данных во Nuxt/Vue
- CORS настроен: разрешены запросы только с доменов учебного заведения
- Студент не имеет доступа к оценкам других студентов

#### 4.4.3 Требования к производительности

| Показатель | Значение |
|-----------|---------|
| Время ответа API (95 перцентиль) | ≤ 500 мс |
| Одновременных пользователей | до 300 |
| Количество записей об оценках в БД | до 500 000 без деградации |

#### 4.4.4 Требования к эргономике

- Адаптивный дизайн: поддержка экранов от 390px (мобильные) до 1920px (desktop)
- Время первой загрузки SPA: ≤ 3 секунды на канале 10 Мбит/с
- Количество кликов для просмотра своих оценок студентом: ≤ 2
- Все интерактивные элементы имеют состояния hover и focus

## 7. ПОРЯДОК КОНТРОЛЯ И ПРИЁМКИ АС

### 7.1 Виды тестирования

| Вид | Описание | Ответственный |
|-----|---------|--------------|
| Модульное | Тестирование отдельных handler-функций Go | Бэкенд-разработчик |
| Интеграционное | Тестирование API через Postman-коллекцию | Бэкенд-разработчик |
| Функциональное | Проверка use cases по тест-плану (UC-01—UC-16) | Весь состав команды |
| E2E / UI | Playwright: визуальные тесты с скриншотами (desktop + mobile) | Фронтенд-разработчик |

### 7.2 Критерии приёмки

Система считается принятой при выполнении всех условий:

- [ ] Все use cases (UC-01 — UC-16) работают без ошибок
- [ ] API возвращает корректные HTTP-коды (200, 201, 400, 401, 403, 404)
- [ ] JWT-аутентификация работает корректно (выдача, проверка, refresh, logout)
- [ ] RBAC ограничения соблюдаются (проверяется попытка доступа с неверной ролью)
- [ ] Студент не может просматривать оценки другого студента
- [ ] История изменений оценки сохраняется при каждом редактировании
- [ ] Интерфейс отображается корректно в Chrome и Firefox
- [ ] Адаптивность на разрешении 390px (мобильное) и 1440px (desktop)
- [ ] Все критичные дефекты устранены
