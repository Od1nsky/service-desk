## 2. ЦЕЛИ И НАЗНАЧЕНИЕ СОЗДАНИЯ СИСТЕМЫ

### 2.1 Цели создания АС

| # | Цель | Критерий достижения |
|---|------|-------------------|
| 1 | Устранить потери заявок | 0% необработанных заявок старше 30 дней |
| 2 | Обеспечить прозрачность обработки | Каждый сотрудник видит статус своей заявки в реальном времени |
| 3 | Сократить среднее время обработки заявки | Среднее время первичного ответа ≤ 4 часов для заявок с приоритетом High |
| 4 | Накапливать историю обращений | Все заявки хранятся в БД бессрочно |
| 5 | Предоставить руководству аналитику | Дашборд с ключевыми показателями (кол-во заявок, среднее время, нагрузка) |

### 2.2 Назначение АС

АС «ServiceDesk» предназначена для автоматизации процесса технической поддержки сотрудников организации: регистрации, маршрутизации, обработки и закрытия заявок на устранение IT-инцидентов и выполнение запросов на обслуживание.

## 3. ХАРАКТЕРИСТИКА ОБЪЕКТА АВТОМАТИЗАЦИИ

### 3.1 Описание объекта

Объект автоматизации — процессы технической поддержки IT-отдела организации, включающие:
- Приём обращений сотрудников
- Классификацию и приоритизацию заявок
- Назначение ответственных специалистов
- Документирование процесса решения
- Контроль соблюдения SLA

### 3.2 Условия эксплуатации

- Система работает в режиме 24/7
- Доступ осуществляется через веб-браузер по корпоративной сети и VPN
- Количество одновременных пользователей: до 100 человек
- Хранение данных: корпоративный сервер или облачная инфраструктура

### 3.3 Пользователи системы

| Категория | Кол-во | Роль в системе |
|-----------|--------|---------------|
| Сотрудники | до 80 | Создание и отслеживание заявок |
| Специалисты поддержки | до 10 | Обработка заявок |
| Администраторы | до 3 | Управление системой |

## 4. ТРЕБОВАНИЯ К АВТОМАТИЗИРОВАННОЙ СИСТЕМЕ

### 4.1 Требования к структуре АС

#### 4.1.1 Состав подсистем

| Подсистема | Назначение |
|-----------|-----------|
| Подсистема аутентификации | Регистрация, вход, управление сессиями (JWT) |
| Подсистема управления заявками | Создание, просмотр, обработка, закрытие заявок |
| Подсистема комментирования | Обмен сообщениями по заявке |
| Подсистема справочников | Управление категориями и приоритетами |
| Подсистема администрирования | Управление пользователями и ролями |
| Подсистема аналитики | Дашборд и статистика |

#### 4.1.2 Архитектура системы

Система реализуется по клиент-серверной архитектуре:

```
┌─────────────────┐     HTTP/REST     ┌──────────────────┐
│  Vue 3 SPA      │ ◄────────────────►│  Go REST API     │
│  (браузер)      │   JSON / JWT      │  (Gin framework) │
└─────────────────┘                   └────────┬─────────┘
                                               │ SQL
                                      ┌────────▼─────────┐
                                      │   PostgreSQL      │
                                      └──────────────────┘
```

#### 4.1.3 Информационное взаимодействие

- Frontend ↔ Backend: REST API, формат JSON, аутентификация Bearer JWT
- Backend ↔ БД: SQL через GORM ORM, соединение по TCP

### 4.2 Требования к функциям (задачам), выполняемым АС

#### 4.2.1 Подсистема аутентификации

| Функция | Результат | Требование |
|---------|-----------|-----------|
| Регистрация пользователя | Учётная запись создана | Email уникален, пароль ≥ 8 символов, хэширование bcrypt |
| Вход в систему | Получен JWT Access + Refresh токен | Access-токен живёт 15 минут, Refresh — 30 дней |
| Обновление токена | Новый Access-токен | Проверка подписи и срока действия Refresh-токена |
| Выход из системы | Refresh-токен инвалидирован | Токен удаляется из БД |

#### 4.2.2 Подсистема управления заявками

| Функция | Результат | Требование |
|---------|-----------|-----------|
| Создание заявки | Заявка зарегистрирована, получен уникальный номер | Обязательные поля: тема, описание, категория, приоритет |
| Просмотр списка заявок | Пагинированный список с фильтрами | Фильтры: статус, приоритет, категория, дата; страница по 20 записей |
| Просмотр деталей заявки | Все данные заявки + история статусов | Доступно автору и специалисту поддержки |
| Назначение заявки | Заявка закреплена за специалистом | Только специалист с ролью support или admin |
| Смена статуса | Статус обновлён, запись в history | Соблюдение допустимых переходов согласно диаграмме состояний |
| Оценка решения | Оценка сохранена | Доступно только автору после перевода в resolved |
| Закрытие заявки | Статус closed, зафиксировано время | Автоматически через 3 дня после resolved или вручную автором |

#### 4.2.3 Подсистема администрирования

| Функция | Результат | Требование |
|---------|-----------|-----------|
| Управление пользователями | Изменение роли / статуса | Только admin; нельзя заблокировать себя |
| Управление категориями | CRUD операции | Нельзя удалить категорию с существующими заявками |
| Просмотр дашборда | Статистика системы | Данные: кол-во по статусам, среднее время решения, топ-5 специалистов |

### 4.3 Требования к видам обеспечения АС

#### 4.3.1 Информационное обеспечение

- СУБД: **PostgreSQL 15+**
- ORM: **GORM v2** (Go)
- Формат обмена данными: **JSON (UTF-8)**
- Структура БД описана в разделе аналитических материалов (файл `03_analytics.md`)
- Резервное копирование БД: ежедневно, хранение 7 резервных копий

#### 4.3.2 Программное обеспечение

**Backend:**
- Язык: **Go 1.21+**
- HTTP framework: **Gin v1.9+**
- ORM: **GORM v2**
- JWT: **golang-jwt/jwt v5**
- Хэширование паролей: **bcrypt** (cost factor ≥ 12)
- Переменные окружения: **godotenv**

**Frontend:**
- Язык: **TypeScript 5+**
- Framework: **Vue 3 (Composition API)**
- Сборщик: **Vite 5+**
- Роутинг: **Vue Router 4**
- Стор: **Pinia 2**
- HTTP-клиент: **Axios**
- UI-библиотека: **Naive UI** или **Element Plus**

**Инфраструктура:**
- Контейнеризация: **Docker + Docker Compose** (опционально)
- ОС сервера: Linux (Ubuntu 22.04 LTS)

#### 4.3.3 Техническое обеспечение

**Минимальные требования к серверу:**
| Параметр | Значение |
|---------|---------|
| CPU | 2 ядра, 2 GHz |
| RAM | 2 GB |
| Диск | 20 GB |
| ОС | Ubuntu 22.04 LTS |
| Сеть | 100 Мбит/с |

**Требования к клиенту (браузер):**
| Параметр | Значение |
|---------|---------|
| Браузер | Chrome 90+, Firefox 88+, Safari 14+, Edge 90+ |
| Разрешение | от 768px (адаптивный дизайн) |
| JavaScript | Включён |

#### 4.3.4 Лингвистическое обеспечение

- Интерфейс системы: **русский язык**
- API-идентификаторы (поля, эндпоинты): **английский язык**
- Кодировка: **UTF-8**

### 4.4 Общие технические требования

#### 4.4.1 Требования к надёжности

| Показатель | Значение |
|-----------|---------|
| Доступность (uptime) | ≥ 99% (не более 7 часов простоя в месяц) |
| Время восстановления после сбоя | ≤ 1 час |
| Сохранность данных при сбое | Транзакционная целостность PostgreSQL |

#### 4.4.2 Требования к безопасности

- Пароли хранятся только в виде хэша bcrypt, не передаются в открытом виде
- HTTPS обязателен в продуктивной среде (TLS 1.2+)
- JWT Access-токен не содержит чувствительных данных, кроме ID и роли
- Все API-запросы (кроме auth) требуют валидного Access-токена
- RBAC (Role-Based Access Control): каждый эндпоинт проверяет роль пользователя
- SQL-инъекции предотвращаются параметризованными запросами через GORM
- XSS предотвращается экранированием выводимых данных во Vue
- CORS настроен: разрешены только запросы с доменов организации

#### 4.4.3 Требования к производительности

| Показатель | Значение |
|-----------|---------|
| Время ответа API (95 перцентиль) | ≤ 500 мс |
| Одновременных пользователей | до 100 |
| Количество заявок в БД | до 100 000 без деградации |

#### 4.4.4 Требования к эргономике

- Адаптивный дизайн: поддержка экранов от 768px (tablet) до 1920px (desktop)
- Время первой загрузки SPA: ≤ 3 секунды на канале 10 Мбит/с
- Количество кликов до создания заявки: ≤ 3
- Все интерактивные элементы имеют состояния hover и focus
